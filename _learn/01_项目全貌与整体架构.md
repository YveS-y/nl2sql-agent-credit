## 内容讲解

### 这个项目解决什么问题？

想象一个场景：业务同学想知道"去年各地区的销售额排名"，但他不会写 SQL，不知道数据库里哪张表叫什么，哪个字段是销售额。

nl2sql-agent 的工作就是：
> 把这句话翻译成一条正确的 SQL，然后把结果返回给他。

但这件事很难，因为 LLM 不知道你的数据库长什么样。解决方案是 **RAG**：先把数据库的元数据（表名、字段名、枚举值）检索出来，再喂给 LLM，让它生成有根据的 SQL。

### 5 层架构

```
用户请求
    │
    ▼
┌─────────────────────────────┐
│  Web 层（FastAPI）           │  接收 HTTP 请求，返回 SSE 流式响应
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│  Server 层（QueryService）   │  调用 Graph，把输出转成 SSE 数据流
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│  Graph 层（LangGraph）       │  12 个节点组成的 Agent 流水线，核心业务逻辑
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│  Repository 层               │  封装数据库读写，打包进 Context 供节点使用
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│  Client 层                   │  管理连接池（MySQL、Qdrant、ES、Embedding）
└─────────────────────────────┘
```

---

### 12 节点流水线

节点按顺序执行，中间有一个条件分支：

```
① extract_keywords     — Jieba 分词，提取名词关键词
        │
   ┌────┴────┐
   ②         ③         ④   ← 三路并行召回
recall_   recall_   recall_
column    value     metric
   └────┬────┘
        │
⑤ merge_retrieved_info — 合并三路结果，去重，补主外键
        │
⑥ filter_table         — LLM 精选相关表和字段
        │
⑦ filter_metric        — LLM 精选相关指标
        │
⑧ add_extra_context    — 注入时间和数据库环境信息
        │
⑨ generate_sql         — LLM 生成 SQL
        │
⑩ validate_sql         — EXPLAIN 测试，不实际执行
        │
    ┌───┴───┐
   错误    正常
    │       │
⑪ correct_sql  ⑫ execute_sql
    │
    └──→ ⑫ execute_sql
```

**关键设计决策：**
- 节点 ②③④ 并行是因为三路召回互不依赖，并行节省时间
- 节点 ⑪ correct_sql 只尝试一次——故意设计的，防止 LLM 陷入死循环
- ⑩ validate_sql 用 `EXPLAIN` 而不是真实执行，避免对数据库产生副作用

### 两类数据结构

理解节点怎么工作，需要先区分两个概念：

| 概念 | 是什么 | 能传什么 |
|---|---|---|
| **State** | 节点间的"共享黑板"，每个节点读它、改它 | 普通 Python 对象（字符串、列表、dict） |
| **Context** | 只读的外部依赖包，节点从这里拿数据库连接 | Repository 对象（不能放进 State） |

类比：State 是会议室白板，Context 是会议室外面的资料柜——白板上的内容每个人都能改，资料柜只能拿不能换。

## 小结

1. **问题本质**：NL → RAG 召回元数据 → LLM 生成 SQL → 执行返回结果
2. **5层架构**：Web → Service → Graph → Repository → Client，每层职责单一
3. **12节点**：分词 → 三路并行召回 → 合并 → 两轮过滤 → 注入上下文 → 生成SQL → 验证 → （纠正）→ 执行
4. **State vs Context**：State 是节点间的共享状态，Context 是只读依赖注入容器
5. **容错设计**：validate + correct 只做一次纠正，不循环，防死循环
