LangGraph 的双通道输出模型
return 管内部协作；writer 管对外广播。两条通道井水不犯河水。
mode	吐什么	谁在用
"values"	每步执行后的完整 state 快照	想观察 state 变化的
"updates"	每个节点 return 的 dict（即通道 A 的内容）	想看增量更新的
"custom"	只吐 writer 主动 push 的东西（即通道 B 的内容）	

text() 干嘛的？	把字符串包装成 SQLAlchemy 的 SQL 对象（TextClause）
为啥必须包？	SQLAlchemy 2.0 起 execute() 禁止接受裸字符串，必须是 SQL 对象
附带作用	支持命名参数绑定（:name 占位），防 SQL 注入的标准姿势

Prompt 软约束 + SQL 硬兜底 + 流式拉数据 + 超时强杀——四道闸门一起上，才叫生产级护栏。
💡 面试加分点 · 为什么"单靠 LIMIT 兜底"不够
很多候选人（包括资深开发）会只说到第 1 层就停，这里补一个能和面试官拉开差距的细节：反例场景：LLM 生成 SELECT COUNT(*) FROM 十亿行大表——
这条 SQL 只返回 1 行（一个数字）
LIMIT 兜底完全不生效（1 ≤ 1000）
但数据库会扫全表，慢查询拖死连接池
能想到这个反例 = 对生产事故有嗅觉。只有第 4 层（超时）能兜住。

 astream 迭代器的"输出通道模式"
writer 只在 custom 通道，values 模式下拿不到

stream_mode	astream 每次 yield 啥	类比
"custom" 🎯 本项目	只吐 writer 主动 push 的东西	节点主动播报的广播
"updates"	节点 return 的增量 dict（本次新改的字段）	每次"改动清单"
"values" ⚠️	每步执行后的完整 state 快照（所有字段）


维度	答
values 模式吐什么	每步执行后的完整 state 快照，所有字段都推
问题 ①	数据爆炸：带宽成本 + 前端 JSON 解析卡顿 + 反向代理可能切断长连接
问题 ②	内部细节泄漏：id/table_id、内部数据模型、原始 SQL、错误栈全部发给前端，安全灾难
设计哲学：
🎯 state 是"内部账本"，只给节点协作看；custom 通道是"对外广播"，主动决定推啥——两者严格分离，正是 LangGraph 双通道设计的精髓。

stream_mode 其实支持列表——

graph.astream(..., stream_mode=["custom", "updates"])
# 每次 yield 的是 (mode_name, chunk) 元组
# 前端自己根据 mode_name 分发
业务场景：想前端同时收到"自定义进度事件" + "调试用的增量变更"——监控页面可能这么用。生产环境不建议开放 values 给前端，但内部 debug/测试时可以临时开。

整体架构分5层，web层，server层，graph层，repository层client层
其中web层通过fastapi联通server层，
graph层分为12个节点，第一个分词节点使用结巴分词，定义好只获取的词性，确保只拿到名词，234节点是3路召回节点，分别去qdrant库和es库拿到分词结果涉及的元数据库能匹配上的字段和指标和值内容信息
5层是合并3个节点召回结果统一整理表-字段-字段内容信息并去重，67层是过通过llm精选字段，8层是拼接时间和数据库环境信息，9层是生成sql，10层是执行计划测试，看看会不会报错，没报错进入12层，报错进入11层，11层是如果报错在这再修正一下，12层是最后正常执行SQL，
其中 23467，11,12 层都使用提示词调用了llm
repository层记录了连接数据库的session信息，合到一个类中，放入context中等待使用，不能传入state，client层是连接各个客户端信息，通过fastapi管理启动和关闭
11层提供了一次纠正问题的机会，但是由于为了不让llm陷入死循环，及时抛出问题，只涉及了一次，
如果要更换支持的业务，只需要修改元数据库中的信息，跑build_meta_knowledge.py 修改meta_config.yaml
其中column/metric都是字段的信息所以召回的时候目的是想让用户问题的切分结果尽量匹配上字段，是语义召回，所以要用qdrant向量召回，value是字段内容，具体的值，所以要严格匹配，通过es匹配召回
yaml对llm更友好，因为占用文本更少，让llm更专注，不会受到格式符号的干扰，同时设置了参数allow_unicode让文字在传递过程中不被转义，同时设定sort_keys不要排序，根据设定好的顺序会让跟重要的信息提前，让llm更好理解，
真正执行sql前肯定要先测试是否成功，并且这个测试不能让garph任务熔断，硬错和软错的区别在于是否可纠正可内部修改，内部借据不了就是硬错，就要上抛，能解决就是软错。并且单节点任务重试如果使用了大模型就要注意不能让人物陷入死循环，做成有界重试
