# NL2SQL Agent — 信贷业务版 项目总结

> 目标：背下来能在面试里自然地讲出这个项目，重点是**为什么这样设计**，不是背代码。

---

## 一、这个项目是什么，解决什么问题

一个让业务同学用自然语言查信贷数据库的 Agent。用户输入"上月各渠道首逾率是多少"，系统自动生成 SQL、执行、返回结果，不需要会写 SQL。

**起点**：已有一个电商版的 NL2SQL Agent（代码已跑通），我的工作是把它完整改造成信贷业务版本，并新增口径澄清和多轮对话两个功能。

---

## 二、系统架构

```
用户 → FastAPI（SSE流式响应）→ LangGraph 工作流 → MySQL（查询结果）
                                     ↑
                               Qdrant（语义召回字段/指标）
                               Elasticsearch（精确召回枚举值）
                               BAAI/bge-large-zh-v1.5（Embedding）
                               DashScope/Qwen（LLM推理）
```

**核心工作流（15个节点，按顺序）**：

```
START
  → rewrite_query（多轮改写）
  → clarify_query（通用清晰度检查）
  → extract_keywords → recall_column / recall_value / recall_metric（并行）
  → merge → filter_table / filter_metric（并行）
  → add_extra_context
  → clarify_metric（歧义指标追问）
  → generate_sql → validate_sql → correct_sql（出错时）→ execute_sql
END
```

**数据仓库分层**（决定 SQL 走哪张表）：
- ADS 层：已聚合的月度指标表，首选，查询代价最低
- DWD 层：30000 条放款明细，只在需要过滤个体记录时才用

## 三、改造做了什么（B1-B7）

### B1-B3：换数据基础

把电商的 5 张表换成信贷的 8 张表（dim_product_type、dim_channel、dim_date、dwd_loan、dwd_repayment、dwd_overdue、ads_loan_monthly、ads_overdue_monthly），写脚本生成 30000 条模拟放款记录，重写元数据配置。

**关键认知**：`meta_config.yaml` 的质量直接决定召回准确率。字段的 `alias`（别名）写得越全，用户用不同词问同一件事时系统越能正确识别。"放款金额"的别名要包括"放款额"、"发放金额"、"贷款金额"等。

### B4：口径澄清节点（clarify_metric + clarify_query）

**问题**：信贷业务里"逾期率"有三种口径——首逾率(M1)、整体逾期率、在贷逾期率。用户说"查一下逾期率"，系统不知道该用哪个，生成的 SQL 可能是错的。

**方案**：在 `add_extra_context` 后插入 `clarify_metric` 节点，用规则检测：如果召回的指标里有"逾期率"且用户没有提供澄清词（首逾/M1/整体/在贷），就追问用户。

**另外**：在 START 后插入 `clarify_query` 节点，用 LLM 判断问题是否连指标方向都没有（如"查一下"），这种情况直接追问，不进入后续流程。

**踩的坑**：`clarify_metric` 最初用"正向触发词"策略（query 里有"逾期率"才追问），但"2026年5月逾期情况怎么样"这种 query 不含"逾期率"却应该追问。改成**负向排除**策略：只有 query 明确含"多少笔/多少条/记录数"这类计数词时才跳过追问，其余一律追问。这样更安全。

### B5：多轮对话（rewrite_query）

**问题**：用户第一轮问"自营APP上月放款额"，第二轮问"抖音呢"，系统不知道"呢"是在问什么。

**方案**：在 START 后第一步插入 `rewrite_query` 节点，把最近 3 轮对话历史传给 LLM，让它把"抖音呢"改写成完整的"抖音上月放款额是多少"再进入后续流程。API 层新增 `conversation_history` 字段，前端维护历史列表。

### B6：Prompt 优化

两个关键规则：
1. **ADS>DWD 分层路由**（写在 filter_table 的 prompt 里）：如果 ADS 层有现成聚合数据，禁止选 DWD 层表，避免在 3 万条明细上做不必要的聚合。
2. **维度聚合规则**（写在 generate_sql 的 prompt 里）：ADS 层有渠道、产品、月份三个维度，如果用户只问渠道，必须 GROUP BY channel_name 并 SUM，不能直接 SELECT，否则每个渠道出现多行。

### B7：30 个测试用例，29/30 通过

分 5 类：简单聚合（8）、多维分析（8）、DWD 明细（4）、口径澄清触发（5）、多轮对话（5）。

遗留问题 [08]：`dwd_loan.loan_date_id` 存的是 yyyyMMdd 整数（如20260531），LLM 生成 SQL 时偶尔用 DATE 类型直接比较，MySQL 隐式类型转换截断到2026，导致空结果。根本解法是把字段改成 DATE 类型，而不是改 prompt。

## 四、最值得讲的三个技术判断

**1. 为什么 clarify_metric 用规则而不是 LLM 判断**

LLM 判断有随机性，今天通过明天可能失败。口径追问是业务强约束——"逾期率"在我们场景里一定要问，不能接受任何概率失败。所以用纯 Python 规则写死，确定性 100%。

**2. 数据格式决定代码，不是代码适应数据格式**

yyyyMMdd 整数日期字段是老数仓遗留设计，LLM 不熟悉这个格式。正确做法是换成 DATE 类型，让 LLM 自然地写日期比较。在字段上打补丁、在 prompt 里写特殊规则，都是治标。

**3. LLM 系统的改动怎么保证安全**

每次改动后跑全量回归测试（30 用例）。修改前先分析哪些已通过的测试可能受影响——这是"负向验证"，比只验证目标修复更重要。我们就是在这里发现了 clarify_metric 改动会破坏"逾期情况怎么样"这个用例。

## 五、一个容易误解的地方：rewrite_query 为什么存在

**误解**：Claude 能处理多轮对话是因为它的 attention 机制更强。

**真相**：attention 是所有现代 LLM 都有的标准机制，不是 Claude 独有的。

Claude 能自然处理"抖音呢"这类指代，是因为**整段对话历史作为输入传给了模型**，模型的 attention 在生成时自然能看到上文。

我们的 NL2SQL Agent 需要 `rewrite_query` 节点，原因是：
- pipeline 里的 `extract_keywords`、`recall_column`、`generate_sql` 等节点只接收 `state["query"]` 这一个字符串
- 历史记录根本没传给这些节点，attention 无从发挥
- 所以需要在 pipeline 入口先把"抖音呢"改写成"抖音上月放款额"，让下游节点拿到完整的查询

**设计原则**：向量召回节点需要干净的单句查询，不适合塞入整段对话。所以在入口集中做一次信息提炼，比给每个节点都传历史更合理。
