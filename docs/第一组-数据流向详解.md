## 一、完整数据流图

```
用户查询 (query)
    ↓
extract_keywords → keywords
    ↓
┌─────────────────────────────────────┐
│         三路并行召回                 │
│  ┌─────┴─────┐                      │
│  ↓     ↓     ↓                      │
│ recall_column  recall_value  recall_metric │
│ (Qdrant)     (ES)          (Qdrant)     │
│  ↓     ↓     ↓                      │
│ retrieved_columns  retrieved_values  retrieved_metrics │
└───────────┬───────────────────────────┘
            ↓
merge_retrieved_info（核心节点）
    ├─ 补充指标依赖字段
    ├─ 补充枚举值字段
    ├─ 补充主外键
    ├─ 按表分组
    └─ Entity → State 转换
            ↓
┌─────────────────────────────┐
│         并行过滤             │
│  ┌─────┴─────┐              │
│  ↓           ↓              │
│ filter_table  filter_metric │
│  ↓           ↓              │
│ table_infos  metric_infos   │
└───────────┬─────────────────┘
            ↓
add_extra_context → date_info, db_info
            ↓
clarify_metric（如需要澄清则结束）
            ↓
generate_sql → sql
            ↓
validate_sql（EXPLAIN验证）
    ├─ 成功 → execute_sql → END
    └─ 失败 → correct_sql → execute_sql → END
```

## 二、State 字段流向表

| 字段 | 来源节点 | 修改节点 | 用途 |
|------|---------|---------|------|
| `query` | 用户输入 | 无 | 所有节点读取 |
| `keywords` | `extract_keywords` | 无 | 三路召回使用 |
| `retrieved_columns` | `recall_column` | 无 | `merge_retrieved_info` 读取 |
| `retrieved_values` | `recall_value` | 无 | `merge_retrieved_info` 读取 |
| `retrieved_metrics` | `recall_metric` | 无 | `merge_retrieved_info` 读取 |
| `table_infos` | `merge_retrieved_info` | `filter_table` | `generate_sql` 读取 |
| `metric_infos` | `merge_retrieved_info` | `filter_metric` | `generate_sql` 读取 |
| `sql` | `generate_sql` | `correct_sql` | `validate_sql`、`execute_sql` |
| `error` | `validate_sql` | 无 | `correct_sql` 读取 |

## 三、核心节点详解

### 1. extract_keywords

```python
# 主要工作
1. jieba 分词提取关键词
2. 保留原始 query（防止过度切分）
3. set 去重
```

### 2. 三路召回

| 节点 | 数据库 | 召回内容 | 目的 |
|------|-------|---------|------|
| `recall_column` | Qdrant | 字段名、别名、描述 | 语义匹配字段 |
| `recall_value` | ES | 枚举值 | 精确匹配 WHERE 条件 |
| `recall_metric` | Qdrant | 指标名、别名、描述 | 语义匹配指标 |

### 3. merge_retrieved_info（最重要）

```python
# 工作顺序
1. 处理指标依赖：遍历 relevant_columns，补充未召回的字段
2. 处理枚举值：通过 column_id 关联到字段，追加到 examples
3. 按表分组：用 dict 做 table_id → columns 映射
4. 补充主外键：支持多表 JOIN
5. 格式转换：Entity（含id）→ State（不含id）
```

### 4. filter_table / filter_metric

```python
# 主要工作
1. 将 table_infos/metric_infos 转成 YAML 格式
2. 调用 LLM，让它判断哪些是需要的
3. 返回过滤后的结果
```

### 5. validate_sql

```python
# 主要工作
1. 执行 EXPLAIN {sql}（不真正执行）
2. 成功：error = None
3. 失败：error = 错误信息
```

## 四、关键实体关联

### ValueInfo → ColumnInfo

```
ValueInfo(
    id="dwd_loan.channel.自营APP",
    value="自营APP",
    column_id="dwd_loan.channel"  # 关联字段
)
            ↓
通过 column_id 找到：
ColumnInfo(
    id="dwd_loan.channel",
    name="channel",
    examples=["自营APP", "京东金融", ...]  # 追加枚举值
)
```

## 五、重要设计决策

### 1. 为什么三路召回要并行？

- 各自独立，没有依赖关系
- 可以同时查询不同数据库（Qdrant/ES）
- 总耗时 = 最慢那路，不是三路之和

### 2. 为什么补充主外键？

- SQL JOIN 需要主键和外键条件
- 不补充的话，LLM 无法写出正确的 JOIN

### 3. 为什么最多修正一次？

- 避免死循环（修正失败 → 再验证 → 再修正）
- 简化设计，以可用性换复杂度

## 六、常见问题

| 问题 | 答案 |
|------|------|
| filter_table 和 filter_metric 直接通信吗？ | 不，都从 merge_retrieved_info 读取 |
| correct_sql 什么时候触发？ | validate_sql 执行 EXPLAIN 失败时 |
| table_infos 被修改几次？ | 两次：merge_retrieved_info 创建，filter_table 修改 |
| 最多几次 SQL 修正机会？ | 1次 |
