# NL2SQL Agent — 迁移到新业务的指南

> 用于把这套框架迁移到其他业务领域（如电商、HR、供应链等）。
> 核心思路：**只换数据和元数据，框架代码尽量不动。**

## 二、meta_config.yaml 的写法

这是影响召回质量最关键的文件，写不好召回就不准。

### 表定义模板

```yaml
tables:
  - name: 表名（英文，如 ads_sales_monthly）
    role: fact          # fact / dim
    layer: ADS          # ADS / DWS / DWD / DIM
    description: >
      一句话说清这张表是什么、有什么数据、何时优先选这张表。
      重要：说明"查询XX类问题时首选本表"，帮助 filter_table 选对表。
    columns:
      - name: 字段名
        role: dimension   # primary_key / foreign_key / dimension / measure
        description: 字段含义，说清枚举值。
        alias: [别名1, 别名2, 别名3]   # 越全越好，决定召回
        examples: [枚举值1, 枚举值2]   # 枚举型字段必填
        sync: true        # true=写入ES，false=不写（枚举值才需要写入ES）
```

### 指标定义模板

```yaml
metrics:
  - name: 指标名（中文，如 销售额）
    description: >
      业务口径说明：怎么算、用哪张表哪个字段、何时用ADS何时用DWD。
    relevant_columns:
      - 表名.字段名
    alias: [别名1, 别名2]
    ambiguous: false   # true=有歧义，需要 clarify_metric 追问
```

### 写好 meta_config 的关键原则

1. **alias 要丰富**：用户可能用"营收"、"收入"、"GMV"指同一件事，全部写进 alias
2. **description 要说清分层**："按月统计时用ADS层，明细分析时用DWD层"
3. **歧义指标加 ambiguous: true**：有多种计算口径的指标必须标记
4. **枚举值字段加 examples**：渠道名、产品名、状态值等，帮助值召回
5. **date 字段用 DATE 类型，不用 yyyyMMdd 整数**：LLM 写 SQL 时自然正确

## 三、clarify_metric.py 的写法

这个文件只需要修改两个地方：

```python
# 1. 歧义指标字典：key=指标名（必须和meta_config里的name一致），value=追问话术
_AMBIGUOUS_METRICS = {
    "逾期率": "您查询的逾期率是指首逾率(M1)、整体逾期率还是在贷逾期率？",
    # 换成新业务的歧义指标，例如：
    # "利润率": "您查询的是毛利率还是净利率？",
    # "销售额": "您查询的是含税还是不含税金额？",
}

# 2. 澄清词：用户说了这些词就不再追问（说明已经澄清了）
_CLARIFICATION_KEYWORDS = {"首逾", "M1", "m1", "整体", "在贷", "综合", "累计"}
# 换成新业务的澄清词，例如：
# {"毛利", "净利", "含税", "不含税", "税前", "税后"}

# 3. 不要改这个！负向排除策略，通用的
_COUNT_QUERY_KEYWORDS = {"多少笔", "笔贷款", "笔逾期", "多少条", "条记录", "记录有多少"}
```

**判断一个指标是否需要 ambiguous: true 的标准**：
- 同一个词有多种计算公式，且不同公式会给出截然不同的数字 → 需要
- 虽然有多种维度切法（按月/按渠道），但公式一样 → 不需要

## 四、clarify_query.prompt 的写法

需要更新两处：

**1. 数据域说明**：换成新业务的指标和维度

```
【数据域说明】
本系统包含以下类型的数据：
- [新业务的指标类型，如"销售数据：GMV、订单量、客单价"]
- [维度，如"维度：地区、品类、门店"]
- [时间粒度说明]
```

**2. 判断规则的示例**：换成新业务的清晰/模糊示例

```
用户：上个月华东区域的GMV是多少 → CLEAR
用户：最近的销售情况 → 您想了解销售额、订单量还是客单价？
```

**不需要改的部分**：
- 判断规则的逻辑（清晰条件/模糊条件）通用
- 输出格式规范（CLEAR vs 一句反问）通用

## 五、filter_table_info.prompt 的分层路由说明

把信贷业务的分层说明换成新业务的，模板如下：

```
【表选择优先级】
1. ADS层表（[新业务ADS表名]）：已预聚合，直接查询，代价最低。
   适合：[新业务的聚合类问题描述]
2. DWD层表（[新业务DWD表名]）：原始明细，仅在以下情况使用：
   - 需要过滤个体记录（如"找出所有XX的用户"）
   - 需要计算ADS层没有的指标

禁止：如果ADS层有满足需求的数据，不得选DWD层表。
```

## 七、常见迁移问题

| 症状 | 原因 | 解决 |
|------|------|------|
| 召回字段/指标不准 | alias 写得少 | 补充更多别名，重建索引 |
| 总是走 DWD 不走 ADS | filter_table prompt 里 ADS 表描述不够清晰 | 在表的 description 里明确写"查询XX问题时首选本表" |
| 歧义指标追问后还是追问 | clarify_metric 的澄清词没有覆盖用户说的词 | 往 `_CLARIFICATION_KEYWORDS` 里加词 |
| SQL 有业务错误（查对表但结果不对） | metric 的 description 没有说清计算口径 | 在 metric description 里写清"计算公式：SUM(X)/COUNT(Y)" |
| LLM 生成 SQL 偶发空结果 | 日期字段类型不对（yyyyMMdd 整数 vs DATE） | 建表时用 DATE 类型，不用整数 |
